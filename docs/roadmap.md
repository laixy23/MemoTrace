# TraceWiki 方案路线

## 24 小时比赛路线

### 0-2 小时：项目骨架

- 建立 Streamlit 应用。
- 建立 `raw/`、`wiki/`、`staging/`、SQLite 存储结构。
- 完成 README 和技术方案文档。

### 2-5 小时：资料摄入

- 文本、Markdown、代码文件解析。
- PDF、Word、表格解析。
- 图片资料保存。

### 5-8 小时：OCR + VLM

- OCR 提取图片文字。
- VLM 生成图片理解。
- 合并为 Image Knowledge Card。

### 8-11 小时：Wiki 知识沉淀

- 自动生成摘要、标签、分类。
- 写入 Markdown Wiki。
- 写入 SQLite 元数据。

### 11-14 小时：问答与证据链

- 检索相关卡片。
- 生成回答。
- 展示 claim-level evidence。

### 14-17 小时：健康审查与补全建议

- 审查缺来源、低覆盖、多模态缺失。
- 判断联网补全还是提醒用户上传。
- 输出知识库健康报告。

### 17-20 小时：内容生成

- 学习笔记。
- 技术报告。
- PPT 大纲。
- Mermaid 思维导图。

### 20-24 小时：演示打磨

- 准备赛题照片、论文、代码样例。
- 完成演示脚本。
- 检查 README 和运行说明。

## 演示脚本

1. 上传赛题照片、论文摘要和代码片段。
2. 展示系统自动生成的 Wiki 卡片。
3. 提问：“这个赛题的核心目标和基础要求是什么？”
4. 展示回答和证据链。
5. 点击“知识库健康审查”。
6. 展示系统发现“图片缺少 OCR/VLM”或“某主题覆盖不足”。
7. 展示补全建议：联网搜索或提醒用户上传。
8. 生成 PPT 大纲和 Mermaid 思维导图。

## 评委讲法

TraceWiki 不是普通 RAG 问答框，而是一个能沉淀、追溯、自检和成长的个人知识库系统。它解决了三个核心问题：

- 用户不知道答案从哪来。
- 知识库不知道自己缺什么。
- 助手不能长期适应用户偏好。

我们的方案分别用证据链、健康审查和用户偏好记忆解决这三个问题。
